# 产品需求文档（PRD）：每日消费记录助手 V1.0

## 1. 文档概述

- **产品名称**：每日消费记录助手
- **版本号**：V1.0（MVP 版本）
- **文档状态**：初稿
- **编写日期**：2026/06/11
- **目标平台**：Web 网页端（响应式设计，优先适配手机浏览器体验）
- **视觉风格**：粉嫩、可爱、治愈系
- **后续规划**：V2.0 迭代原生 App（iOS/Android），V3.0 覆盖微信小程序，实现多端数据同步。

## 2. 产品定位与背景

### 2.1 产品背景
随着日常消费场景的多样化（微信支付、支付宝、现金等），个人用户往往难以清晰掌握自己的资金流向。市面上现有的记账软件功能过于繁杂，界面偏商务冷峻，广告较多，导致年轻用户记账动力下降。

### 2.2 产品定位
打造一款**极简、轻量、无打扰、超可爱**的个人 Web 端记账工具。用户无需下载 App，打开手机浏览器即可使用。通过粉嫩的视觉语言和流畅的交互，让记账这件“反人性”的事变得轻松治愈。

### 2.3 目标用户
- **核心用户**：18-30 岁年轻群体，尤其是喜欢可爱风格、有日常记账习惯或希望培养记账习惯的个人用户。
- **使用场景**：每日消费后 30 秒内快速补记；月末回顾消费构成；偶尔导出数据备份。

## 4. 非功能需求

### 4.1 性能需求
- **首屏加载**：在 4G 网络下，首屏加载时间不超过 2 秒。
- **交互响应**：用户点击、滑动操作反馈延迟不超过 100ms。
- **数据容量**：本地存储（IndexedDB）至少支持 5 年内每日 20 笔记录的存储空间而不出现明显卡顿。

### 4.2 兼容性需求
- **浏览器兼容**：
  - 移动端：iOS Safari (iOS 13+)、Chrome for Android (Android 9+)、微信内置浏览器。
  - PC 端：Chrome、Edge、Safari 最新两个主版本。
- **屏幕适配**：采用响应式布局，核心适配 375px - 430px 宽度的手机屏幕，同时兼容平板和 PC 端浏览（居中卡片式布局，限制最大宽度 600px）。

### 4.3 安全与隐私
- **数据存储**：V1.0 所有数据仅存储在用户浏览器本地（IndexedDB）。**不上传任何数据至服务器**。
- **数据风险告知**：在首次引导和“个人中心”中，明确提示用户“清理浏览器缓存将导致数据丢失，建议定期导出备份”。

### 4.4 可用性需求
- **离线可用**：支持 PWA（Progressive Web App）基础配置，允许用户“添加到主屏幕”，在无网络环境下也能正常记账和查看。
- **无登录态**：V1.0 不强制要求用户注册登录，打开即用。

## 5. 视觉与风格规范

### 5.1 设计方向
本产品面向年轻个人用户，主打**粉嫩、可爱、治愈**的视觉感受。界面应避免冷峻、商务、复杂的设计语言，通过柔和的配色、大圆角卡片和圆润的图标传递轻松愉悦的记账心情。

### 5.2 配色方案

| 色彩角色 | 色值（HEX） | 使用场景 |
| :--- | :--- | :--- |
| **主色调（樱花粉）** | `#FFB7C5` | 顶部导航背景、核心按钮背景、选中状态高亮 |
| **强调色（奶油桃）** | `#FFD1DC` | 次级按钮、标签背景、图表高亮区块 |

| **背景色（米白渐变）** | `#FFF5F7` → `#FFFFFF` | 页面全局背景，顶部至底部轻微粉色渐变 |
| **卡片底色（纯白）** | `#FFFFFF` | 内容卡片、输入框背景 |
| **主文字色（深灰棕）** | `#5D4037` | 标题、金额数字、重要正文 |
| **次要文字色（暖灰）** | `#8D7B7B` | 日期、辅助说明、Placeholder |

| **收入色（薄荷绿）** | `#A5D6A7` | 收入金额、成功状态、正向反馈（V1.0 预留） |

### 5.3 UI 风格细节
- **圆角规范**：所有卡片、按钮、输入框统一使用大圆角，**16px - 24px**。禁止直角或小圆角（4px）。
- **阴影风格**：使用柔和的弥散阴影（`box-shadow: 0 4px 20px rgba(255, 183, 197, 0.25)`），避免生硬的深色投影。
- **图标风格**：采用 **线面结合** 的圆润图标，描边粗细统一为 2px，端点为圆角。避免尖锐的几何图形。鼓励使用 Emoji 作为分类图标选项之一。
- **插画元素**：空状态、引导页、记账成功反馈中，使用扁平化的小动物（如小猫、小熊、兔子）、星星、云朵、爱心等元素。
- **字体建议**：正文使用系统默认无衬线字体（如 `-apple-system, BlinkMacSystemFont, "Segoe UI"`）。金额数字推荐使用等宽或圆润感强的字体，保证对齐且亲切。

### 5.4 动效反馈
- 点击按钮时，使用轻微的缩放动画（`scale(0.96)` → `scale(1)`），配合弹性缓动（cubic-bezier）。
- 记账成功时，顶部飘出粉色爱心或星星粒子动效，给予正向反馈。
- 切换 Tab 时，底部图标使用弹性缓动上下跳动，增加活泼感。
- 页面切换使用左右滑动转场（模拟 App 导航体验）。

## 6. 数据模型（前端本地存储）

### 6.1 消费记录 (Record)

```typescript
interface Record {
  id: string;          // 唯一标识，UUID
  amount: number;      // 金额，单位元，保留两位小数
  type: 'expense';     // V1.0 仅支持 expense，数据结构预留 'income' 供后续版本扩展
  categoryId: string;  // 关联的分类 ID
  remark: string;      // 备注，默认空字符串
  date: string;        // 日期，格式 YYYY-MM-DD
  createdAt: number;   // 创建时间戳，用于排序
  updatedAt: number;   // 更新时间戳
}
```

### 6.2 消费分类 (Category)

```typescript
interface Category {
  id: string;          // 唯一标识
  name: string;        // 分类名称，如 "餐饮"
  icon: string;        // 图标名称/Emoji（如 "🍔" 或 Iconfont 名称）
  type: 'expense';     // V1.0 仅支持 expense，预留 'income'
  isCustom: boolean;   // 是否为用户自定义添加（系统预设为 false）
  createdAt: number;   // 创建时间戳
}
```

### 6.4 数据交换规范（导入/导出）

导出的 `.json` 文件需符合以下 Schema：

```json
{
  "version": "1.0",
  "exportDate": "2026-06-11T10:00:00Z",
  "records": [
    {
      "id": "uuid-string",
      "amount": 36.00,
      "type": "expense",
      "categoryId": "cat_dining",
      "remark": "午餐",
      "date": "2026-06-11",
      "createdAt": 1718092800000,
      "updatedAt": 1718092800000
    }
  ],
  "categories": [
    {
      "id": "cat_dining",
      "name": "餐饮",
      "icon": "🍔",
      "type": "expense",
      "isCustom": false,
      "createdAt": 1718092800000
    }
  ]
}
```

**导入容错机制**：
- 若 `version` 不兼容，提示用户“文件版本过旧/过新，无法导入”。
- 若单条记录字段缺失（如缺少 `amount`），跳过该条并提示“已跳过 X 条异常记录”。
- 导入前必须要求用户二次确认，明确告知将**覆盖**现有本地数据。

## 8. 技术要求与约束

- **前端框架**：建议采用 React + Vite 或 Vue 3 + Vite 进行开发，配合 TypeScript。
- **UI 组件库**：移动端优先，建议使用 Vant (Vue) 或 Ant Design Mobile (React)，以便快速实现底部弹层、日期选择器等。
- **图表库**：统计页甜甜圈图建议使用 ECharts 或 Chart.js。
- **本地存储**：优先使用 IndexedDB（通过 `localForage` 或 `Dexie.js` 封装）存储消费记录；简单配置项可用 LocalStorage。
- **构建与部署**：构建产物为纯静态文件，可部署至 GitHub Pages / Vercel / Netlify 或国内静态网站托管，无需后端服务器。

## 10. 风险与应对

| 风险点 | 描述 | 应对措施 |
